# Project Working Preferences

이 프로젝트에서 작업할 때는 아래 도구와 워크플로를 우선 고려한다.

- Context7: 라이브러리, 프레임워크, API 사용법이 작업에 영향을 주면 최신 문서와 예제를 확인한다.
- Playwright MCP: 브라우저 동작, UI 회귀, E2E 흐름, 스크린샷 기반 검증이 필요하면 우선 사용한다.
- GitHub: 이슈, PR, Actions, 릴리스, 원격 저장소 상태가 작업에 필요하면 GitHub 도구를 사용한다. 원격 변경은 명시적 요청 없이 수행하지 않는다.
- Sentry: 운영 오류, 성능 문제, 사용자 영향도 확인이 필요하면 Sentry 도구로 근거를 확인한다. 민감한 값은 노출하지 않는다.
- Skill: 작업과 맞는 Codex Skill이 있으면 해당 `SKILL.md`를 먼저 읽고 지침을 따른다.
- codex-action: 자동화나 CI 기반 Codex 작업이 필요하면 기존 GitHub Actions 패턴을 우선 확인하고, 관련 워크플로는 검증 가능한 형태로 유지한다.

## Multi-agent 작업 방식

범위가 넓은 작업은 바로 나누기보다 먼저 2~3가지 분할 방식을 비교한다. 기능별, 계층별, 위험도별 분할 중 병렬성이 높고 충돌이 적은 방식을 고른다.

- 여러 agent가 동시에 작업할 때는 각 agent의 소유 파일 또는 책임 범위를 명확히 지정한다.
- explorer agent는 코드베이스 질문과 맥락 조사에, worker agent는 독립적인 코드 변경에, 검증 agent는 병렬 검증에 사용한다.
- 동일 파일을 여러 agent가 동시에 수정해야 하는 분할은 피한다. 불가피하면 메인 agent가 최종 통합과 일관성 검토를 맡는다.
- 각 agent 결과는 근거, 변경 범위, 검증 방법을 짧게 남기고, 최종 응답에서는 병렬 작업의 결론과 남은 위험만 합쳐서 보고한다.
- agent가 생성한 변경은 다른 agent의 작업을 되돌리지 않고 현재 작업 트리 상태에 맞춰 통합한다.

요청한 도구가 현재 세션에서 사용할 수 없거나 설치되어 있지 않으면, 가능한 범위를 먼저 알리고 가장 가까운 안전한 대안을 사용한다. 도구 사용 여부를 추측해서 말하지 않는다.
